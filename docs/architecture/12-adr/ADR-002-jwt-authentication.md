# ADR-002: JWT認証の採用

## ステータス

**Accepted** - 2025-01-08

## コンテキスト

mobile-app-system では、以下の要件があります：

1. **マルチクライアント**: モバイルアプリ（iOS/Android）と管理Webアプリの認証
2. **ステートレス**: サーバー側でセッション状態を保持したくない
3. **スケーラビリティ**: 複数のサーバーインスタンスで動作可能にしたい
4. **セキュリティ**: 十分なセキュリティレベルを確保

認証方式として、以下の選択肢があります：

- セッションベース認証（Cookie）
- トークンベース認証（JWT）
- OAuth 2.0 / OpenID Connect

## 決定

**JWT（JSON Web Token）トークンベース認証を採用する**

### 仕様

- **アルゴリズム**: HS256（HMAC with SHA-256）
- **トークン有効期限**: 24時間
- **リフレッシュトークン**: なし（デモ用途）
- **保存場所**:
  - モバイル: Keychain（iOS）/ EncryptedSharedPreferences（Android）
  - Web: localStorage（デモ用途、本番ではHttpOnly Cookie推奨）

### JWTペイロード構造

```json
{
  "sub": "1",                    // ユーザーID
  "loginId": "user001",          // ログインID
  "userType": "user",            // ユーザー種別（user/admin）
  "iat": 1704700800,             // 発行日時
  "exp": 1704787200              // 有効期限（24時間後）
}
```

## 理由

### JWTのメリット

1. **ステートレス**
   - サーバー側でセッション状態を保持不要
   - 複数サーバーインスタンスで動作可能

2. **スケーラビリティ**
   - セッションストアが不要
   - 水平スケーリングが容易

3. **モバイル対応**
   - モバイルアプリでの使用に適している
   - Cookieを使用しないため、Webviewでも動作

4. **自己完結型**
   - トークン内にユーザー情報を含む
   - DBアクセス不要で検証可能

### デメリットと対策

| デメリット | 対策 |
|----------|------|
| トークン無効化が困難 | 有効期限を短く設定（24時間） |
| トークンサイズが大きい | 必要最小限の情報のみペイロードに含む |
| XSS脆弱性（Web） | 本番ではHttpOnly Cookie推奨 |

## 代替案

### 代替案1: セッションベース認証（Cookie）

**メリット**:
- セッション無効化が容易
- サーバー側で完全制御

**デメリット**:
- ステートフル（セッションストアが必要）
- スケーラビリティに課題
- モバイルアプリで使いにくい

**却下理由**: モバイルアプリとの相性が悪い、スケーラビリティに課題

### 代替案2: OAuth 2.0 / OpenID Connect

**メリット**:
- 標準化された認証プロトコル
- 外部IdP連携可能

**デメリット**:
- 実装が複雑
- デモ用途にはオーバースペック
- 外部IdP不要

**却下理由**: デモ用途には複雑すぎる、外部IdP連携の必要なし

## 結果

### 期待される効果

1. **シンプルな実装**: 標準ライブラリ（jjwt）で実装可能
2. **高いパフォーマンス**: DBアクセス不要で検証高速
3. **モバイル対応**: モバイルアプリで使いやすい

### 実装への影響

- **Web API**: JWT生成・検証ロジックの実装
- **クライアント**: トークンをセキュアストレージに保存
- **セキュリティ**: シークレットキーを環境変数で管理

### セキュリティ考慮事項

1. **シークレットキー管理**
   - 環境変数で管理
   - 最小256ビット（32文字以上）

2. **トークン保存**
   - iOS: Keychain使用
   - Android: EncryptedSharedPreferences使用
   - Web: localStorage（本番ではHttpOnly Cookie推奨）

3. **HTTPS必須**
   - 本番環境ではHTTPS通信必須
   - 開発環境はHTTP許可

### 制限事項（デモ用途）

- リフレッシュトークンなし（有効期限後は再ログイン）
- トークン無効化機能なし
- 多要素認証なし
