# ADR-001: BFFパターンの採用

## ステータス

**Accepted** - 2025-01-08

## コンテキスト

mobile-app-system では、以下の要件があります：

1. **複数のクライアント**: iOS、Android、管理Webアプリの3種類のクライアントが存在
2. **クライアント固有のニーズ**: 各クライアントで必要なデータ形式・API構造が異なる可能性
3. **保守性**: Web APIとクライアントの疎結合を維持したい
4. **段階的拡張**: 将来的にクライアント固有のロジックを追加する可能性

従来のアーキテクチャでは、クライアントが直接Web APIを呼び出す構成が一般的ですが、
以下の課題があります：

- クライアント種別ごとに異なるデータ構造が必要な場合、Web APIが複雑化
- クライアント固有のエラーハンドリングやリトライロジックが各クライアントに分散
- Web APIの変更がすべてのクライアントに直接影響

## 決定

**BFF（Backend For Frontend）パターンを採用する**

以下の2つのBFFを構築します：

1. **Mobile BFF** (Port: 8081)
   - iOS/Androidアプリ専用
   - モバイル最適化されたAPI提供

2. **Admin BFF** (Port: 8082)
   - 管理Webアプリ専用
   - 管理機能最適化されたAPI提供

### アーキテクチャ

```
[iOS/Android] --> [Mobile BFF] --> [Web API] --> [SQLite]
[Admin Web]   --> [Admin BFF]  --> [Web API] --> [SQLite]
```

### BFFの責務

- Web APIへのリクエスト中継
- エラーハンドリング
- タイムアウト管理
- クライアント固有のレスポンス最適化（将来拡張）

### BFFで実装しないこと

- ビジネスロジック（Web APIに集約）
- データベースアクセス（Web APIに集約）
- JWT検証（Web APIで実施）

## 理由

### メリット

1. **責務の分離**
   - クライアント固有のロジックをBFFに集約
   - Web APIはビジネスロジックに専念

2. **保守性の向上**
   - Web API変更時、BFFで吸収可能
   - クライアントへの影響を最小化

3. **柔軟性**
   - クライアントごとに最適化されたAPI提供
   - 将来の拡張が容易（キャッシュ、集約等）

4. **セキュリティ**
   - BFFで認証トークンを中継
   - Web APIへの直接アクセスを防止

### デメリットと対策

| デメリット | 対策 |
|----------|------|
| レイヤーが増え複雑化 | BFFはシンプルに保つ（中継のみ） |
| 運用コストの増加 | Docker化で運用を簡略化（将来） |
| パフォーマンス影響 | BFFのオーバーヘッドは最小限（< 10ms） |

## 代替案

### 代替案1: クライアントから直接Web API呼び出し

**メリット**:
- シンプルなアーキテクチャ
- レイテンシが少ない

**デメリット**:
- クライアント固有のロジックが分散
- Web API変更の影響が大きい

**却下理由**: 将来の拡張性と保守性を考慮し、BFFパターンを優先

### 代替案2: GraphQLの採用

**メリット**:
- クライアントが必要なデータのみ取得
- 柔軟なクエリ

**デメリット**:
- 学習コスト高
- デモ用途にはオーバースペック

**却下理由**: デモ用途にはRESTful APIで十分

## 結果

### 期待される効果

1. **スケーラビリティ**: クライアント追加時にBFFを追加するのみ
2. **保守性**: Web APIとクライアントの疎結合
3. **柔軟性**: クライアント固有の要件に対応可能

### 実装への影響

- **開発工数**: BFF実装のため+2週間
- **運用**: BFFコンテナの追加（+2プロセス）
- **インフラ**: ポート8081, 8082の開放

### モニタリング

- BFF-Web API間のレスポンス時間を監視
- BFFのエラー率を監視
