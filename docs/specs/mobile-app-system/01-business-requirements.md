## 1. ビジネス目的

### 1.1 プロジェクトの背景

**BR-001: デモンストレーション用途**
- **背景**: 現代的なモバイルアプリケーション開発のベストプラクティスを示す
- **目的**: BFFパターン、マイクロサービス、機能フラグなどの技術アーキテクチャの実証
- **期待効果**: 技術的な意思決定の参考となるサンプル実装の提供

**BR-002: アーキテクチャパターンの実証**
- **目的**: 以下のアーキテクチャパターンの有効性を実証
  - Backend For Frontend (BFF) パターン
  - マイクロサービスアーキテクチャ
  - JWT認証
  - 機能フラグによる段階的リリース

### 1.2 ビジネスゴール

| ID | ゴール | 説明 | 優先度 |
|----|-------|------|--------|
| BG-001 | 技術実証 | モダンなアプリケーションアーキテクチャの実証 | 高 |

## 2. ビジネス要件

### 2.1 エンドユーザー向け要件

#### 2.1.1 ユーザーログイン

**BR-010: ログイン機能**
- **要件**: ユーザーはID/パスワードでログインできる
- **ビジネス価値**: ユーザーの識別と個別機能の提供
- **受入基準**:
  - 正しい認証情報でログインできる
  - ログイン後、商品一覧画面に遷移する
  - 誤った認証情報の場合、エラーメッセージが表示される
  - JWTトークンが発行される

**BR-011: セッション管理**
- **要件**: ログイン状態は一定期間保持される
- **ビジネス価値**: ユーザー体験の向上（再ログイン頻度の削減）
- **受入基準**:
  - トークン有効期限内はログイン状態が維持される
  - 有効期限切れ時は再ログインが必要

#### 2.1.2 商品閲覧・検索

**BR-020: 商品一覧表示**
- **要件**: ログイン後、ユーザーは商品一覧を閲覧できる
- **ビジネス価値**: 商品の可視化と選択の起点
- **受入基準**:
  - 全商品が一覧表示される
  - 商品名、単価、基本情報が表示される
  - リスト形式で見やすく表示される

**BR-021: 商品検索**
- **要件**: ユーザーは商品名で商品を検索できる
- **ビジネス価値**: 目的の商品への迅速なアクセス
- **受入基準**:
  - キーワードで部分一致検索できる
  - 検索結果が即座に表示される
  - 該当商品がない場合、その旨が表示される

**BR-022: 商品詳細表示**
- **要件**: ユーザーは商品の詳細情報を閲覧できる
- **ビジネス価値**: 購入判断に必要な情報の提供
- **受入基準**:
  - 商品名、単価、説明文が表示される
  - 購入ボタンが表示される
  - お気に入りボタンが表示される（機能フラグON時）

#### 2.1.3 商品購入

**BR-030: 100個単位での購入**
- **要件**: ユーザーは商品を100個単位でのみ購入できる
- **ビジネス価値**: 業務ルールの実装（まとめ買い前提）
- **ビジネスルール**:
  - 購入個数は100、200、300...と100の倍数のみ
  - 1〜99個など100の倍数以外は購入不可
- **受入基準**:
  - 100個単位の数量選択UIが提供される
  - 選択した個数と単価から合計金額が計算される
  - 購入確定で注文が記録される

**BR-031: 購入記録**
- **要件**: 購入情報はシステムに記録される
- **ビジネス価値**: 購入履歴の保持（将来的な分析や履歴表示に活用可能）
- **受入基準**:
  - ユーザーID、商品ID、個数、購入日時が記録される
  - 購入完了メッセージが表示される

**BR-032: 決済処理の除外**
- **要件**: 実際の決済処理は行わない（デモ用途）
- **ビジネス理由**: デモンストレーション用のため
- **受入基準**:
  - 購入ボタン押下で即座に購入完了となる
  - 決済画面や外部連携は発生しない

#### 2.1.4 お気に入り機能

**BR-040: お気に入り登録**
- **要件**: ユーザーは商品をお気に入りに登録できる（機能フラグ有効時）
- **ビジネス価値**: ユーザーエンゲージメントの向上
- **受入基準**:
  - 機能フラグがONのユーザーのみ機能が表示される
  - お気に入り登録/解除ができる
  - お気に入り状態が保持される

**BR-041: 機能フラグによる制御**
- **要件**: お気に入り機能はユーザー単位でON/OFF制御される
- **ビジネス価値**: 段階的な機能リリース、A/Bテストの実現
- **ビジネスルール**:
  - 機能フラグOFFのユーザーには機能が表示されない
  - 機能フラグは管理者が動的に変更できる
  - 変更は即座に反映される
- **受入基準**:
  - フラグOFFユーザーはお気に入りボタンが非表示
  - フラグONユーザーはお気に入りボタンが表示される
  - フラグ変更後、次回API呼び出しで反映される

### 2.2 管理者向け要件

#### 2.2.1 管理者ログイン

**BR-050: 管理者ログイン**
- **要件**: 管理者は専用のログイン画面からログインできる
- **ビジネス価値**: 管理機能へのアクセス制御
- **ビジネスルール**:
  - 管理者アカウントはエンドユーザーアカウントと分離
  - 管理者専用の認証情報を使用
- **受入基準**:
  - 管理者用ID/パスワードでログインできる
  - 管理者用JWTトークンが発行される
  - ログイン後、管理ダッシュボードに遷移する

#### 2.2.2 商品管理

**BR-060: 商品情報編集**
- **要件**: 管理者は商品の名前と単価を変更できる
- **ビジネス価値**: 商品情報の柔軟な管理
- **ビジネスルール**:
  - 商品名は必須、1〜100文字
  - 単価は正の整数、1円以上
  - 変更は即座に反映される
- **受入基準**:
  - 商品一覧が表示される
  - 商品の編集画面で名前と単価を変更できる
  - 保存後、変更が反映される
  - モバイルアプリに即座に反映される

**BR-061: 商品追加・削除の除外**
- **要件**: 初期実装では商品の追加・削除は行わない
- **ビジネス理由**: スコープの限定（デモ用途）
- **受入基準**:
  - 商品の新規追加機能は提供されない
  - 商品の削除機能は提供されない
  - 既存商品の編集のみ可能

#### 2.2.3 機能フラグ管理

**BR-070: ユーザー別機能フラグ設定**
- **要件**: 管理者はユーザー単位で機能フラグをON/OFFできる
- **ビジネス価値**: 段階的機能リリース、A/Bテストの実現
- **ビジネスルール**:
  - 初期実装は「お気に入り機能」フラグのみ
  - ユーザー単位で個別に設定可能
  - デフォルト値は全ユーザーOFF
- **受入基準**:
  - ユーザー一覧が表示される
  - 各ユーザーの機能フラグ状態が表示される
  - ON/OFFトグル操作ができる
  - 変更が即座に保存される

**BR-071: 機能フラグの即時反映**
- **要件**: 機能フラグの変更は即座にモバイルアプリに反映される
- **ビジネス価値**: リアルタイムな機能制御
- **受入基準**:
  - 変更後、ユーザーの次回API呼び出しで反映される
  - アプリ再起動は不要

## 3. ユースケース

### 3.1 エンドユーザーのユースケース

#### UC-001: 商品を検索して購入する

**アクター**: エンドユーザー

**前提条件**:
- ユーザーアカウントが存在する
- 商品データが登録されている

**基本フロー**:
1. ユーザーはアプリを起動する
2. ログイン画面でID/パスワードを入力する
3. ログインボタンをタップする
4. 商品一覧画面が表示される
5. 検索ボックスに商品名を入力する
6. 検索結果から目的の商品をタップする
7. 商品詳細画面が表示される
8. 購入個数を選択する（100, 200, 300...）
9. 合計金額を確認する
10. 購入ボタンをタップする
11. 購入完了メッセージが表示される

**代替フロー**:
- 2a. 認証情報が不正な場合、エラーメッセージが表示される
- 5a. 該当商品が見つからない場合、その旨が表示される
- 8a. 購入個数を変更すると合計金額が再計算される

**事後条件**:
- 購入記録がDBに保存される

**ビジネスルール**:
- BR-010, BR-020, BR-021, BR-022, BR-030, BR-031

#### UC-002: お気に入りに商品を追加する

**アクター**: エンドユーザー（機能フラグON）

**前提条件**:
- ユーザーがログイン済み
- 当該ユーザーの「お気に入り機能」フラグがON

**基本フロー**:
1. ユーザーは商品詳細画面を表示する
2. お気に入りボタンが表示される
3. お気に入りボタンをタップする
4. ボタンの状態が「お気に入り済み」に変わる
5. お気に入り登録完了メッセージが表示される

**代替フロー**:
- 3a. 既にお気に入り済みの場合、タップでお気に入り解除される

**事後条件**:
- お気に入り情報がDBに保存される

**ビジネスルール**:
- BR-040, BR-041

### 3.2 管理者のユースケース

#### UC-010: 商品の単価を変更する

**アクター**: 管理者

**前提条件**:
- 管理者アカウントでログイン済み
- 変更対象の商品が存在する

**基本フロー**:
1. 管理者は管理画面にログインする
2. 商品管理メニューを選択する
3. 商品一覧が表示される
4. 編集したい商品の編集ボタンをクリックする
5. 商品編集画面が表示される
6. 単価フィールドに新しい単価を入力する
7. 保存ボタンをクリックする
8. 変更完了メッセージが表示される
9. 商品一覧に戻り、変更が反映されている

**代替フロー**:
- 6a. 不正な値（負の数、文字列等）を入力した場合、バリデーションエラーが表示される

**事後条件**:
- 商品の単価がDBで更新される
- モバイルアプリで新しい単価が表示される

**ビジネスルール**:
- BR-060

#### UC-011: ユーザーの機能フラグを変更する

**アクター**: 管理者

**前提条件**:
- 管理者アカウントでログイン済み
- 対象ユーザーが存在する

**基本フロー**:
1. 管理者は機能フラグ管理メニューを選択する
2. ユーザー一覧が表示される
3. 各ユーザーの現在のフラグ状態が表示される
4. 変更したいユーザーのフラグトグルをクリックする
5. フラグ状態が切り替わる（ON⇔OFF）
6. 変更が自動保存される
7. 変更完了の通知が表示される

**事後条件**:
- ユーザーの機能フラグ設定がDBで更新される
- 対象ユーザーの次回API呼び出しで新しいフラグ状態が反映される

**ビジネスルール**:
- BR-070, BR-071

## 4. ビジネスルール

### 4.1 認証・認可ルール

| ルールID | ルール内容 |
|---------|-----------|
| BIZ-R-001 | エンドユーザーと管理者は別の認証基盤を使用する |
| BIZ-R-002 | 未ログインユーザーは商品一覧にアクセスできない |
| BIZ-R-003 | エンドユーザーは管理者用APIにアクセスできない |
| BIZ-R-004 | 管理者はエンドユーザー用APIにアクセスできない |
| BIZ-R-005 | JWTトークンにユーザー種別情報を含める |

### 4.2 商品ルール

| ルールID | ルール内容 |
|---------|-----------|
| BIZ-R-010 | 商品名は1〜100文字 |
| BIZ-R-011 | 商品単価は1円以上の整数 |
| BIZ-R-012 | 商品には一意のIDが割り当てられる |
| BIZ-R-013 | 商品の削除は行わない（論理削除も含む） |

### 4.3 購入ルール

| ルールID | ルール内容 |
|---------|-----------|
| BIZ-R-020 | 購入個数は100の倍数のみ |
| BIZ-R-021 | 最小購入個数は100個 |
| BIZ-R-022 | 最大購入個数は9900個（制限値） |
| BIZ-R-023 | 合計金額 = 単価 × 個数 |
| BIZ-R-024 | 在庫チェックは行わない |
| BIZ-R-025 | 実際の決済処理は行わない |

### 4.4 機能フラグルール

| ルールID | ルール内容 |
|---------|-----------|
| BIZ-R-030 | 機能フラグはユーザー単位で設定される |
| BIZ-R-031 | デフォルトの機能フラグ状態はOFF |
| BIZ-R-032 | 機能フラグOFFの場合、該当機能は非表示 |
| BIZ-R-033 | 初期実装では「お気に入り機能」フラグのみ |
| BIZ-R-034 | 機能フラグ変更は即座に反映される |

### 4.5 お気に入りルール

| ルールID | ルール内容 |
|---------|-----------|
| BIZ-R-040 | 1ユーザーあたりお気に入り登録数の上限はなし |
| BIZ-R-041 | 同じ商品を重複してお気に入り登録できない |
| BIZ-R-042 | お気に入り解除は即座に反映される |
| BIZ-R-043 | 機能フラグOFFの場合、既存のお気に入りデータは保持される |

## 7. 制約事項

### 7.1 ビジネス制約

**BC-001: デモ用途限定**
- 本番環境への展開は想定しない
- 実際の商取引は行わない
- 個人情報保護は簡易的な対応

**BC-002: 機能スコープの限定**
- 決済機能は実装しない
- 在庫管理は実装しない
- 注文履歴表示は実装しない
- パスワードリセット機能は実装しない

**BC-003: ユーザー登録機能の除外**
- ユーザーはマスタデータとして事前登録
- 自己登録機能は提供しない

### 7.2 時間的制約

**TC-001: デモンストレーションまでの期限**
- 具体的な期限は未定（Phase 1完了時点）
- 段階的な開発・デモが可能

## 8. 前提条件

### 8.1 ビジネス前提

**BA-001: ターゲットユーザー**
- エンドユーザー: スマートフォンの基本操作ができる方
- 管理者: システム管理の基本知識を持つ方

**BA-002: 利用環境**
- インターネット接続が常時利用可能
- iOS 15以降、Android 10以降を想定

**BA-003: データ準備**
- 初期ユーザーデータは開発側で準備
- 初期商品データは開発側で準備
- サンプルデータはSQL初期化スクリプトで投入
