# リスクと対策

## 1. 技術リスク

### 1.1 Struts2 2.3.x のセキュリティ脆弱性（リスク：**極高**）

| 項目 | 内容 |
|---|---|
| リスク概要 | Struts2 2.3.x は Apache によるセキュリティサポートが終了している。OGNL インジェクション等の重大な脆弱性が過去に多数報告されており、今後新たな脆弱性が発見されてもパッチは提供されない |
| 影響 | リモートコード実行（RCE）、情報漏洩、サービス停止 |
| 対策 | ① OGNL 式の動的評価を無効化する設定を適用 ② WAF（Web Application Firewall）の導入 ③ ネットワークセグメンテーションで外部からの直接アクセスを制限 ④ 定期的な脆弱性スキャンの実施 |
| 代替案 | Struts2 2.5.x（Java 8 必須）の採用を検討。JDK 1.7 制約の緩和が可能であれば最も効果的な対策 |

### 1.2 JDK 1.7 のサポート終了（リスク：**高**）

| 項目 | 内容 |
|---|---|
| リスク概要 | JDK 1.7（Oracle）は 2022 年 3 月にプレミアサポートが終了。セキュリティアップデートは提供されない |
| 影響 | 既知のセキュリティホールが修正されないまま運用される |
| 対策 | ① OpenJDK 7 の長期サポート版（Azul Zulu 等）を使用 ② TLS 1.2 対応の確認 ③ セキュリティパッチの有無を定期確認 |
| 代替案 | JDK 8 への変更を強く推奨（Struts2 2.5.x も利用可能になる） |

### 1.3 SQLite 同時アクセス競合（リスク：**中**）

| 項目 | 内容 |
|---|---|
| リスク概要 | SQLite はファイルベースの DB であり、複数プロセス（Struts2 アプリ + 既存 Web API）からの同時書き込みでロック競合が発生する可能性がある |
| 影響 | DB ロックエラー（SQLITE_BUSY）、データ不整合 |
| 対策 | ① SQLite の WAL（Write-Ahead Logging）モードを有効化 ② JDBC 接続時に `busy_timeout` を設定（例：5000ms） ③ 書き込み処理のリトライロジック実装 ④ コネクションプーリングの適切な設定 |

### 1.4 ライブラリ互換性問題（リスク：**中**）

| 項目 | 内容 |
|---|---|
| リスク概要 | Java 7 互換のライブラリバージョンが古く、既知のバグや互換性の問題がある可能性 |
| 影響 | ビルド失敗、実行時エラー、予期しない動作 |
| 対策 | ① Phase 0 で全依存関係の互換性を事前検証 ② Maven の `enforcer` プラグインで Java バージョンチェック ③ 代替ライブラリの事前調査 |

## 2. プロジェクトリスク

### 2.1 UI/UX 品質低下（リスク：**高**）

| 項目 | 内容 |
|---|---|
| リスク概要 | Vue 3 SPA → JSP MPA への移行により、ユーザー体験が大幅に低下する |
| 具体例 | ページ遷移ごとの画面リロード、トグルボタンのリアルタイム更新不可、フォームバリデーションのレスポンス低下 |
| 対策 | ① jQuery / AJAX による部分的な非同期更新の導入 ② CSS フレームワーク（Bootstrap 3 等）による見た目の品質確保 ③ FeatureFlag のトグルは AJAX で更新する設計 |

### 2.2 開発生産性低下（リスク：**中**）

| 項目 | 内容 |
|---|---|
| リスク概要 | Java 7 の言語制約（ラムダ/Streams 不可）、JSP 開発のフィードバックループの遅さ |
| 対策 | ① JSP ホットデプロイ設定（Tomcat の reloadable 設定） ② IDE の Struts2 サポートプラグインの導入 ③ DAO/Service のテスト駆動開発 |

### 2.3 既存 Web API との JSON 互換性（リスク：**中**）

| 項目 | 内容 |
|---|---|
| リスク概要 | Struts2 Web API が返す JSON 形式が既存 Spring Boot Web API と完全に一致しない可能性 |
| 影響 | 内部システムが Struts2 API に接続した場合のパースエラー |
| 対策 | ① 既存 API のレスポンスを JSON Schema として形式化 ② 自動比較テストの作成 ③ カスタム JSON シリアライザの実装 |

## 4. 推奨事項

本移行計画の実施にあたり、以下を強く推奨する：

1. **JDK バージョンの再検討**: JDK 1.7 → JDK 8 への変更が許容される場合、Struts2 2.5.x（アクティブサポート）が利用可能になり、セキュリティリスクが大幅に低減する

2. **段階的移行**: admin-bff と admin-web を同時に廃止するのではなく、まず Struts2 管理画面を構築・安定稼働させてから切り替える

3. **ロールバック計画**: 移行後に重大な問題が発生した場合に備え、既存の admin-web / admin-bff を即時復旧できる状態を維持する

4. **セキュリティ監査**: Struts2 2.3.x を採用する場合、デプロイ前に第三者によるセキュリティ監査を実施する
