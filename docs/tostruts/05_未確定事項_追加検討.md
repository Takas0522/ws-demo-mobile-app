# 未確定事項・追加検討事項

> **前提**: 本アプリケーションは PoC（Proof of Concept）として構築され、Public な Web サイトとして公開されない。  
> そのため、本番環境向けのセキュリティ対策やスケーラビリティの要件は緩和される。  
> 以下の各項目では、PoC であることを踏まえた現実的な判断を記載する。

## 1. データベースアクセスに関する未確定事項

### 1.1 SQLite の同時アクセス制御

**現状**: 既存の Web API（Spring Boot, port 8080）は HikariCP コネクションプールを使用して SQLite にアクセスしている。WAL（Write-Ahead Logging）モードの設定は **行われていない**。

**課題**: Struts2 アプリ（port 8082）が同一の `src/web-api/data/mobile_app.db` に直接アクセスする場合、2 つの独立した JVM プロセスが同一の SQLite ファイルを共有することになる。

| 検討項目 | 選択肢 | 備考 |
|---|---|---|
| WAL モード有効化 | ① DB 初期化スクリプトで `PRAGMA journal_mode=WAL;` を実行 ② 各アプリの起動時に PRAGMA を発行 | WAL モードにより読み取り/書き込みの並行性が向上する。PoC では ① が簡便 |
| busy_timeout 設定 | JDBC URL に `?busy_timeout=5000` を付与 | ロック競合時に即時エラーではなく 5 秒間リトライする |
| DB ファイルパスの共有方式 | ① 環境変数 `DB_PATH` で統一 ② Struts2 側の `application.properties` にハードコード | PoC では ② で十分だが、起動スクリプトとの整合性に注意 |

**要決定**: WAL モードを有効化するか、busy_timeout をどの程度に設定するか。

**管理者による記載**: 競合を起こさない使い方になるため不要

### 1.2 コネクション管理方式

**現状**: 技術設計方針では `DatabaseUtil` + `Commons DbUtils` による手動コネクション管理を想定している。

**課題**: コネクションプーリングの有無が未定義。

| 選択肢 | メリット | デメリット |
|---|---|---|
| プーリングなし（都度接続） | 実装がシンプル | SQLite はファイルロックなので大きな性能差はない |
| Commons DBCP 1.4 | Java 7 互換のプーリング | 設定が増える |
| Tomcat JNDI DataSource | Tomcat の設定で管理 | context.xml の設定が必要 |

**PoC 推奨**: SQLite は軽量なファイル DB であり、PoC レベルの同時接続数であればプーリングなし（都度接続 + `try-with-resources` で即時クローズ）で十分と考えられる。

**管理者による記載**: 記載の通りプーリングなしで十分

### 1.3 トランザクション管理

**現状**: 既存 Web API では Spring の `@Transactional` アノテーションで宣言的トランザクション管理を行っている。

**課題**: Struts2 + JDBC 直接アクセスでは、トランザクション境界を明示的に管理する必要がある。

**要決定**:
- Service 層でトランザクションを手動管理するか（`conn.setAutoCommit(false)` → `conn.commit()` / `conn.rollback()`）
- トランザクション管理のユーティリティクラスを作成するか
- 特に商品更新時の **価格履歴自動記録**（既存の SQLite トリガー `trg_record_price_history`）がトランザクション内で正しく動作するかの確認

## 2. 認証・セッション管理に関する未確定事項

### 2.1 管理画面の認証方式（セッション vs JWT）

**現状の計画**: 管理画面はセッション認証、Web API は JWT 認証の二重構成を想定。

**課題**: 2 つの認証方式を同一 WAR 内で共存させる設計の詳細が未確定。

| 検討項目 | 詳細 |
|---|---|
| セッションストア | Tomcat のインメモリセッション（デフォルト）で十分か |
| セッションタイムアウト | 30 分（web.xml で設定済み）で適切か |
| 管理画面の CSRF 対策 | PoC のため省略可能。ただし Struts2 にはトークンインターセプタ（`<s:token>`）がある |
| JWT の共有 | Struts2 API が発行する JWT は既存 Web API と同じ秘密鍵を使用するか。同じ鍵を使えばトークに互換性が得られるが、鍵管理の仕組みが必要 |

**PoC 推奨**: 管理画面はセッション認証のみで十分。Struts2 Web API の JWT は既存 Web API と同一の `JWT_SECRET` 環境変数を参照する設計とし、トークン互換性を確保する。

### 2.2 既存パスワードハッシュとの互換性

**現状**: 既存 DB のパスワードは `BCryptPasswordEncoder(strength=10)` でハッシュ化されている（Spring Security 実装）。シードデータの例:
```
$2a$10$9XQy779t3Zn.q2vGeMn6we/cHZw5nDdKd7oD1hsn8Li5TWRKPD3HC
```

**課題**: Struts2 アプリで `jBCrypt 0.4` を使用した場合、Spring Security が生成したハッシュと互換性があるかの検証が必要。

**確認事項**: jBCrypt の `BCrypt.checkpw(plaintext, hash)` が Spring Security の bcrypt ハッシュを正しく検証できるか。bcrypt 自体は標準仕様のため互換性があると見込まれるが、Phase 0 で実際に検証する。

### 2.3 JWT ライブラリの Java 7 互換バージョン

**現状**: 既存 Web API は `jjwt 0.12.3` を使用。この バージョンは Java 8 以降が必要。

**課題**: Java 7 互換の JJWT `0.6.0` は API が大きく異なり、`SignatureAlgorithm`、`Claims` の操作方法が変わる。

**要確認**: 
- JJWT 0.6.0 で HS256 署名の生成・検証が動作するか
- 既存 Web API が発行した JWT トークンを Struts2 側で検証できるか（逆も同様）
- 秘密鍵の形式（Base64 エンコード）に差異がないか

## 9. 既存 DB トリガーとの互換性

### 9.1 価格履歴の自動記録

**現状**: SQLite に `trg_record_price_history` トリガーが定義されており、`products` テーブルの `unit_price` が UPDATE されると自動的に `product_price_history` に履歴が INSERT される。

```sql
CREATE TRIGGER trg_record_price_history
AFTER UPDATE OF unit_price ON products
WHEN OLD.unit_price != NEW.unit_price
BEGIN
    INSERT INTO product_price_history (product_id, old_price, new_price, changed_at, changed_by, change_reason)
    VALUES (NEW.product_id, OLD.unit_price, NEW.unit_price, CURRENT_TIMESTAMP, NEW.updated_by, 'Price updated');
END;
```

**課題**: 
- `changed_by` は `NEW.updated_by` を参照しているが、`products` テーブルに `updated_by` カラムが存在するか確認が必要
- Struts2 側の UPDATE SQL で `updated_by` を適切に設定しないと、トリガーが正しく動作しない可能性がある
- 既存 Web API（JPA）側で `updated_by` をどのように設定しているかの確認が必要

### 9.2 updated_at 自動更新トリガー

**現状**: `users`, `products`, `user_feature_flags` テーブルに `updated_at` 自動更新トリガーがある。

**確認事項**: JDBC 直接の UPDATE 文でもこれらのトリガーは発火するため、Struts2 側で `updated_at` を明示的に SET する必要はない。ただし `SELECT` で取得した値とトリガー更新後の値にタイムスタンプの差異が生じうることに留意。

## 11. PoC 特有の考慮事項

### 11.1 セキュリティリスクの再評価

前回のリスク分析で指摘した Struts2 2.3.x の OGNL インジェクション脆弱性や JDK 1.7 の EOL について、**PoC かつ非公開環境** であることを踏まえて再評価する。

| リスク | 本番想定時 | PoC（非公開）時 | 対応方針 |
|---|---|---|---|
| Struts2 OGNL インジェクション | **極高** — RCE 攻撃の標的 | **低** — 外部からのアクセスなし | PoC では許容。将来の本番化時には Struts2 2.5+ への移行が必須 |
| JDK 1.7 EOL | **高** — セキュリティパッチ不在 | **低** — 内部ネットワーク限定 | PoC では許容 |
| CSRF 未対策 | **高** | **低** — 利用者が限定的 | PoC では省略可能 |
| セッション固定攻撃 | **中** | **低** | PoC では標準のセッション管理で十分 |
| SQLite ファイルアクセス権 | **中** | **低** | OS レベルのファイルパーミッションで十分 |

### 11.2 スコープの最適化案

PoC の目的が「Struts2 での管理画面構築と Web API 提供の技術検証」であれば、スコープを更に絞ることが可能。

| 項目 | フルスコープ | PoC 最小スコープ | 削減効果 |
|---|---|---|---|
| 管理画面 | 6画面すべて | ログイン + 商品一覧 + 商品編集 の 3 画面 | 画面数 50% 削減 |
| Struts2 Web API | 19 エンドポイント | 認証 + 商品 CRUD の 7 エンドポイント | エンドポイント 63% 削減 |
| DAO 層 | 7 テーブル分 | users + products + product_price_history の 3 テーブル | DAO 57% 削減 |
| テスト | DAO + Service + Action + API | DAO のみ | テスト工数 75% 削減 |

### 11.3 将来的な本番移行時の追加要件

PoC から本番に移行する場合、以下が追加で必要となる：
- JDK 8 以降へのアップグレードと Struts2 2.5+ への移行
- OGNL インジェクション対策の適用
- CSRF トークンの実装
- HTTPS の有効化
- セッション管理の強化（HttpOnly, Secure フラグ）
- 監査ログの強化
- 入力値のサニタイズ強化

| # | 項目 | 関連フェーズ | 優先度 |
